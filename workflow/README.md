# rental_car Workflow

> 本目录是 rental_car 的开发执行入口。所有实现任务必须先锚定 `WF-*` 任务包，再开始代码改动。

## 1. 目录结构

```text
workflow/
├─ README.md
├─ index.md
├─ owners/
│  └─ unassigned.md
└─ tasks/
   ├─ P0/
   ├─ P1/
   └─ P2/
```

## 2. 核心原则

- `workflow/index.md` 是任务编排、优先级、依赖和状态的唯一真相源。
- 每个任务包粒度为 1-2 天，并对应一个可独立验收的业务闭环。
- owner 只表示主责人，不表示前后端边界；谁领取谁负责接口、页面、联调、自测和文档同步。
- 当前 owner 统一为 `待定`，认领后再更新为具体开发人员。
- 不按 C 端、管理端、PC 后台或后端拆散同一个业务闭环。
- P0 是首期 MVP 主链路；P1 是首期增强；P2 是二期 backlog。

## 3. 状态流转

- `待开始`：依赖未满足，不能认领。
- `可领取`：依赖已满足，可进入开发。
- `开发中`：当前 owner 正在实现。
- `待验收`：开发完成，等待按任务包验收。
- `已完成`：验收通过，可向下游 handoff。
- `阻塞`：已开始但因依赖、外部条件或需求冲突无法继续。
- `待规划`：仅用于 P2 backlog。

默认流转：`待开始 -> 可领取 -> 开发中 -> 待验收 -> 已完成`。

依赖判定：用户确认后，前置任务处于 `待验收` 也可视为下游开发已解锁；下游完成前仍需回看前置任务是否存在影响接口、页面或配置的验收问题。

## 4. 状态同步硬规则

任何状态变化都必须同步更新：

1. `workflow/index.md`
2. 当前任务包 `workflow/tasks/P0|P1|P2/<task>.md`
3. 对应 owner 视图 `workflow/owners/<owner>.md`

任务结束必须更新开发 plan 进度。未完成上述三处同步，不得声称任务完成。

## 5. 任务包固定字段

每个任务包必须包含以下字段，顺序固定：

1. `目标`
2. `业务闭环`
3. `输入文档`
4. `范围内改动`
5. `非目标`
6. `前置依赖`
7. `交付物`
8. `验收动作`
9. `完成定义`
10. `下游 handoff`
11. `当前状态`

所有任务包必须明确：`owner 负责该任务全栈闭环，不再拆前后端子任务。`

## 6. 使用方式

1. 从 `workflow/index.md` 查看可领取任务。
2. 打开任务包，按范围、交付物和验收动作实施。
3. 开始编码前把状态切到 `开发中`。
4. 开发结束后按验收动作验证，并把状态切到 `待验收` 或 `已完成`。
5. 如遇阻塞，写清阻塞点、已尝试动作、缺少条件和建议处理人。
