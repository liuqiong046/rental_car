# 山海放心租 API 接口规范 V1

## 1. 目标

本规范定义 rental_car 后端 API 的统一形态。接口风格参考 onePet 的 FastAPI 单仓多模块实现，但字段、资源和业务语义全部按租车 PRD 定制。

## 2. 路由与模块

所有业务接口统一使用 `/api/v1` 前缀。模块按业务资源拆分：

| 模块 | 路由前缀 | 说明 |
| --- | --- | --- |
| auth | `/api/v1/auth` | 微信登录、手机号登录、刷新会话、退出 |
| users | `/api/v1/users` | 当前用户、身份认证、黑名单提示 |
| cities | `/api/v1/cities` | 开通城市、首页配置、服务规则 |
| vehicles | `/api/v1/vehicles` | 车辆列表、详情、库存状态、车辆管理 |
| prices | `/api/v1/prices` | 价格日历、费用预估、价格规则 |
| orders | `/api/v1/orders` | 客户订单、锁车、支付后履约状态 |
| wholesale-orders | `/api/v1/wholesale-orders` | 独立批发订单、关联批发订单 |
| work-orders | `/api/v1/work-orders` | 工单、排班、车况、现场费用 |
| payments | `/api/v1/payments` | 支付、回调、退款、流水 |
| finance | `/api/v1/finance` | 押金、结算、批发成本、利润 |
| admin | `/api/v1/admin` | 组织、账号、角色、审计、导出 |
| messages | `/api/v1/messages` | 站内信、订阅消息记录 |

路由命名使用资源名复数，动作优先体现为状态迁移子资源：

- `POST /api/v1/orders/{order_id}/cancel`
- `POST /api/v1/orders/{order_id}/dispatch`
- `POST /api/v1/wholesale-orders/{wholesale_order_id}/accept`
- `POST /api/v1/work-orders/{work_order_id}/condition-records`

### 2.1 C 端车辆浏览接口

`GET /api/v1/vehicles` 用于 C 端首页和车辆列表，返回已审核、已上架、非维保、非锁车、未占用且所选日期有可租价格的车辆。

支持筛选参数：

- `city_code`
- `pickup_at` / `return_at`：ISO 本地时间字符串，例如 `2026-05-06T10:00:00`
- `brand`
- `price_min` / `price_max`
- `color`
- `vehicle_type`
- `energy_type`
- `seats`
- `gearbox`

公开车辆响应只允许返回 `plate_mask`，不得返回完整车牌号。公开价格只返回 C 端价、可租状态和可租时段，不返回底价和批发价。

`GET /api/v1/vehicles/{vehicle_id}` 用于 C 端车辆详情，除车辆公开字段外返回 `price_calendar`，供详情页展示价格日历。不可租车辆返回 `409` 和稳定 `detail`，例如 `车辆维保中`、`车辆已锁定`、`当前日期缺少可租价格`。

### 2.2 C 端确认订单与未支付锁车接口

`POST /api/v1/orders/estimate` 用于确认订单页费用预估，需要 C 端登录。请求包含车辆、城市、取还车时间、取还车方式、地址摘要和备注。响应返回租金、上门送收服务费、优惠、应支付金额、押金提示、违章押金提示、价格快照和城市未支付锁车分钟数。

`POST /api/v1/orders` 用于创建待支付订单并短时锁车，需要 C 端登录、身份认证通过和 `Idempotency-Key` Header。创建成功返回 `201`，订单状态为 `pending_payment`，支付状态为 `unpaid`，并返回 `lock_expires_at`。未认证用户返回 `403` 和 `身份认证通过后才能提交订单`。

`POST /api/v1/orders/admin/expire-unpaid` 用于后台或定时任务关闭超时未支付订单并释放车辆，需要后台登录。该接口首期用于验证和任务调度占位；正式落库后应由调度器调用并记录操作日志。

确认订单接口不得返回手机号、证件号、驾驶证号或详细地址原文；C 端只传和展示 `pickup_address_summary` / `return_address_summary`。

### 2.3 C 端支付预下单与微信回调接口

`POST /api/v1/payments/prepay` 用于 C 端对待支付订单创建微信支付预下单，需要 C 端登录和 `Idempotency-Key` Header。请求包含 `order_id`。订单必须属于当前用户，且订单状态为 `pending_payment`、支付状态为 `unpaid`；支付他人订单返回 `403` 和 `只能支付本人订单`，非法状态返回 `409` 和 `当前订单状态不允许支付`。

首期支付 provider 为 `wechat_mock`，响应返回 `payment_id`、`order_id`、`provider`、`payable_amount`、`currency`、`status`、`pay_params`、`mock_callback_url` 和 `created_at`。`pay_params` 只包含前端调起支付需要的临时参数，不包含商户密钥、证书、API key 或支付配置明文。

`POST /api/v1/payments/wechat/callback` 用于处理微信支付回调，需要 `Idempotency-Key` Header。首期 mock/sandbox 请求包含 `order_id`、`payment_id`、`transaction_id`、`trade_state` 和 `paid_amount`。`SUCCESS` 回调会校验支付流水和订单金额一致，并将订单推进为 `order_status=pending_acceptance`、`payment_status=paid`；重复回调返回稳定结果且 `processed=false`，不会重复写流水或重复推进订单状态。金额不一致返回 `400` 和 `支付金额不一致`。

支付成功后车辆占用继续保留，不能被旧未支付订单清理释放；后续 `WF-P0-11` 从 `pending_acceptance` 承接客户订单接单、改派和退单。

### 2.4 PC 后台客户订单接单、改派与退单接口

`GET /api/v1/orders/admin/customer-orders` 用于 PC 后台查询已支付客户订单，需要后台登录。可用 `status` 筛选订单状态，列表统一返回 `{ items, total }`。当前首期返回支付后订单，不返回手机号、证件号、驾驶证号或详细地址原文。

`GET /api/v1/orders/admin/customer-orders/{order_id}` 用于查询客户订单详情，需要后台登录。响应包含订单金额、支付状态、订单状态、取还车摘要、车辆来源、改派差价结果、待退单原因和操作日志。

`POST /api/v1/orders/admin/customer-orders/{order_id}/accept` 用于运营中心接单，需要后台登录和 `Idempotency-Key` Header。只有 `payment_status=paid` 且 `order_status=pending_acceptance` 的订单允许接单；自有/托管车辆接单后进入 `pending_dispatch`。车行车辆在 `WF-P0-12` 之前不能直接接单，返回 `409` 和 `车行车辆需创建关联批发订单`。

`POST /api/v1/orders/admin/customer-orders/{order_id}/reassign` 用于改派同车型车辆，需要后台登录和 `Idempotency-Key` Header。请求包含 `target_vehicle_id` 和可选 `remark`。改派成功后订单进入 `pending_dispatch`，记录 `original_vehicle_id`、`reassign_price_difference` 和 `reassign_result`。新车价格高于原订单时差价结果为 `运营中心承担差价`，低于原订单时为 `待结算退差`。

`POST /api/v1/orders/admin/customer-orders/{order_id}/return` 用于无法履约时进入待退单，需要后台登录和 `Idempotency-Key` Header。请求包含 `reason`。成功后订单进入 `pending_return`，退款审核和原路退款归属 `WF-P0-18`。

### 2.5 批发订单创建、接拒单与超时拒绝接口

`GET /api/v1/wholesale-orders` 用于 PC 后台和后续管理端查询批发订单，需要后台登录。可用 `status` 筛选，列表统一返回 `{ items, total }`。总部和运营中心按数据范围查看，车行只查看本车行批发订单。

`POST /api/v1/wholesale-orders/admin/related` 用于运营中心为车行客户订单创建关联批发订单，需要后台登录和 `Idempotency-Key` Header。请求包含 `customer_order_id` 和可选 `remark`。仅允许 `payment_status=paid` 且 `order_status=pending_acceptance` 的车行车辆客户订单创建；成功后返回 `wholesale_order_id`、`customer_order_id`、`dealer_id`、`vehicle_id`、`wholesale_price`、`status=pending_dealer_acceptance` 和 `expires_at`。

`POST /api/v1/wholesale-orders/{wholesale_order_id}/change-price` 用于车行修改批发价，需要后台登录和 `Idempotency-Key` Header。请求包含 `wholesale_price` 和 `reason`。只有 `pending_dealer_acceptance` 批发订单可改价，重复提交返回首次结果。

`POST /api/v1/wholesale-orders/{wholesale_order_id}/accept` 用于车行接单，需要后台登录和 `Idempotency-Key` Header。成功后批发订单进入 `accepted`，关联客户订单进入 `pending_dispatch`。

`POST /api/v1/wholesale-orders/{wholesale_order_id}/reject` 用于车行拒单，需要后台登录和 `Idempotency-Key` Header。请求包含 `reason`。成功后批发订单进入 `rejected`，关联客户订单进入 `pending_reassign`，并记录 `批发订单已拒绝：<reason>`。

`POST /api/v1/wholesale-orders/admin/expire-pending` 用于后台或定时任务执行车行确认 SLA 超时拒绝，需要后台登录。首期可传 `minutes_after_sla` 作为测试和任务调度占位；正式落库后应由调度器按城市 `dealer_confirm_sla_minutes` 调用。超时后批发订单进入 `expired_rejected`，关联客户订单进入 `pending_reassign`。

## 3. 后端文件分层

每个业务模块默认结构：

```text
backend/app/apps/<module>/
├─ router.py    # APIRouter、路径、依赖、response_model
├─ schemas.py   # Pydantic v2 请求与响应 DTO
├─ service.py   # 业务编排、状态迁移、事务边界
├─ deps.py      # 鉴权、权限、数据范围依赖，必要时新增
└─ models.py    # SQLAlchemy 模型，落地数据库任务时新增
```

规则：

- `router.py` 不写复杂业务，只做参数接收、依赖注入和调用 service。
- `schemas.py` 的响应模型不得暴露明文敏感字段。
- `service.py` 负责事务、状态机、幂等、审计、外部调用编排。
- 跨模块共享的错误、鉴权、日志、配置放在 `app/core/`。

## 4. FastAPI 路由约定

每个路由必须声明：

- `response_model`
- `summary`
- 创建类接口的 `status_code=201`
- 删除类接口的 `status_code=204`
- 需要登录、角色、数据范围时使用 `Depends(...)`

示例：

```python
router = APIRouter(prefix="/api/v1/orders", tags=["orders"])

@router.post(
    "",
    response_model=OrderDetail,
    status_code=status.HTTP_201_CREATED,
    summary="创建客户订单并锁车",
)
async def create_order(
    payload: OrderCreate,
    idempotency_key: str = Depends(require_idempotency_key),
    user_id: str = Depends(require_current_user_id),
) -> OrderDetail:
    return await order_service.create_order(user_id, payload, idempotency_key)
```

## 5. DTO 与响应结构

请求/响应 DTO 使用 Pydantic v2。字段命名默认 snake_case，与 Python 和后端模型保持一致；小程序和后台适配层负责前端命名转换。

列表响应统一：

```json
{
  "items": [],
  "total": 0
}
```

分页参数统一：

- `page`: 从 1 开始
- `page_size`: 默认 20，最大 100
- `keyword`: 模糊搜索
- 其他筛选字段按资源语义命名

敏感字段响应规则：

- 手机号只返回 `phone_mask`
- 身份证只返回 `id_no_mask`
- 驾驶证只返回脱敏编号或图片授权 URL
- 详细地址按角色返回 `address_summary` 或经审计授权的明文
- 支付配置、密钥、token、cipher 不得进入响应体

## 6. 鉴权、权限与数据范围

- C 端、管理端、PC 后台都必须由服务端校验身份。
- 前端隐藏按钮不能作为权限控制。
- 总部可看全平台；运营中心默认只看本城市；车行只看本车行及授权协同订单；司机只看本人任务。
- 鉴权依赖放在 `deps.py` 或 `app/core/auth/`。
- 数据范围过滤必须在查询层或 service 层统一执行，禁止只在前端筛选。

建议依赖：

- `require_current_user_id`
- `require_admin_actor`
- `require_role(*roles)`
- `require_data_scope(resource, action)`
- `require_idempotency_key`

## 7. 幂等规则

以下接口必须要求 `Idempotency-Key` Header：

- 创建客户订单
- 支付预下单
- 支付回调处理
- 退款发起
- 客户订单接单、退单、改派
- 批发订单接单、拒绝、修改价格
- 工单完成
- 现场费用新增
- 押金/免押确认
- 结算完成

服务端 24 小时内对同一 actor、同一接口、同一 key 返回首次处理结果。业务已完成但重复提交时返回稳定结果，不重复写流水、不重复推进状态。

## 8. 错误返回

首期沿用 FastAPI 默认错误结构：

```json
{
  "detail": "请先登录"
}
```

基建任务可扩展为统一错误码，但必须兼容 `detail`：

```json
{
  "code": "ORDER_STATUS_INVALID",
  "detail": "当前订单状态不允许接单",
  "request_id": "req_xxx"
}
```

常用状态码：

- `200`: 查询、更新成功
- `201`: 创建成功
- `204`: 删除或无响应体成功
- `400`: 业务参数不合法或状态不允许
- `401`: 未登录
- `403`: 无权限或数据范围不允许
- `404`: 资源不存在
- `409`: 幂等冲突、状态冲突、库存冲突
- `422`: DTO 校验失败

## 9. OpenAPI 与接口评审

- FastAPI OpenAPI 是三端协作的接口事实来源。
- 新增接口必须有 `summary`、tags、请求/响应 DTO。
- 修改接口必须同步本规范或对应 `workflow/tasks/*` 的接口契约说明。
- 高风险接口在任务包验收动作中必须包含状态码、响应字段、鉴权、幂等和失败路径验证。

## 10. 接口测试规范

接口测试参考 onePet：

- 使用 `pytest` + `pytest-asyncio` + `httpx.AsyncClient`。
- 测试文件放在 `backend/tests/apps/<module>/test_*_api.py`。
- 覆盖状态码、关键响应字段、鉴权失败、数据范围隔离、非法状态迁移、幂等重复提交。
- expected 在前，actual 在后；断言字段用业务语义，不只断言 200。

示例场景：

- 未登录访问订单列表返回 401。
- 未认证用户不能进入支付。
- 未支付订单超时后关闭并释放车辆。
- 车行拒绝关联批发订单后，客户订单进入待改派/待退单。
- 重复支付回调只推进一次订单状态。
- 无权限车行不能查看其他车行批发订单。
